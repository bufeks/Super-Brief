# Super-Brief

> 広告ブリーフ・リサーチに特化した、Claude Code のための「最高の人材」装備一式。
> ブリーフを渡すと、競合・市場・社会の空気・生活者・カルチャーを複合的にリサーチし、
> プランニングチームが最高のスタートダッシュを切れる HTML レポートを納品する。

> 🔒 **このリポジトリは「設定」のみ**を保持します。
> ブリーフ入力 (PDF / PPTX / DOCX) と生成 HTML レポートは **git にコミットしません**。
> 案件成果物はリポジトリの外で管理し、共有する場合は HTML を直接送ってください。

---

## できること

- PDF / PPTX / DOCX / テキストのブリーフを読み解く
- 10 領域フレームワーク (競合 / 市場 / 社会の空気 / 生活者 / カルチャー / リスク 等) で網羅リサーチ
- WebSearch / WebFetch を使った一次情報優先のリサーチ
- 領域横断のテンション・矛盾の発見
- プランナーへの引き継ぎカード (仮説の種) を 3〜5 個
- 単一ファイルで自己完結する HTML レポートを `reports/` に出力

**やらないこと**: 広告プラン (コンセプト、コピー、媒体プラン) の提示。リサーチに特化。

---

## 使い方

### 1. リポジトリを開く

このリポジトリを Claude Code で開く (CLI / Web / Desktop / IDE どれでも)。

### 2. 依存パッケージのインストール (初回のみ)

```bash
pip install -r requirements.txt
```

PDF / PPTX / DOCX を抽出するために必要。テキストや Markdown のみ扱う場合は不要。

### 3. ブリーフを渡す

#### パターン A: ファイルを添付

ブリーフ資料 (`.pdf`, `.pptx`, `.docx`) を Claude Code セッションに添付して、

```
このブリーフをリサーチして
```

と送るだけ。Claude が自動で `research-brief` フローを起動する。

#### パターン B: テキストを貼る

ブリーフの中身をそのまま貼り付けて送る。

#### パターン C: 明示的にスキル呼び出し

```
/research-brief
```

### 4. レポートを受け取る

`reports/<YYYYMMDD>-<案件名スラッグ>.html` に出力される。
ブラウザで開けば、外部 CSS/JS なしで読める単一ファイル HTML。印刷にも対応。

## ディレクトリ構成

```
Super-Brief/
├── README.md
├── CLAUDE.md                 # リサーチャー人格＋自動起動ルール
├── requirements.txt
├── .claude/skills/research-brief/SKILL.md   # /research-brief フロー本体
├── scripts/extract.py        # ブリーフ→テキスト変換
├── templates/
│   ├── report.html           # HTML レポートのスタイル＆構造テンプレート
│   └── framework.md          # 10 領域フレームワーク詳細
└── reports/                  # 生成レポート出力先 (中身は git 管理外)
```

ブリーフ入力ファイルは **このリポジトリの中に置かない**。`/tmp/<案件名>.pdf` のような
リポジトリ外の場所に置き、絶対パスで `scripts/extract.py` に渡してください。

## リサーチの 10 領域

| # | 領域 | 何を見るか |
|---|---|---|
| 1 | ブリーフ・デコード | 表層課題 / 真因仮説 / 違和感 |
| 2 | 競合状況 | 直接・間接・新興・越境競合 |
| 3 | 市場・カテゴリー | 規模・成長率・構造変化・規制 |
| 4 | 社会の空気 (Zeitgeist) | 価値観の潮流・バズ・文化的瞬間 |
| 5 | 生活者インサイト | N1 の声・Job to be done・Tension |
| 6 | ブランド現在地 | 認知・好意度・過去キャンペーン |
| 7 | カルチャー文脈 | サブカルチャー・ミーム・影響力者 |
| 8 | 機会領域 (White Space) | 未充足ニーズ・隣接カテゴリー |
| 9 | リスク・地雷 | 炎上・規制・ブランドセーフティ |
| 10 | 仮説の種 | プランナーへの引き継ぎカード |

詳細は [`templates/framework.md`](templates/framework.md) を参照。

## 品質基準

レポートは以下を満たしていなければ Claude が自分で再修正する:

- [ ] ブリーフのデコードが冒頭にある
- [ ] 10 領域のうち最低 6 領域カバー
- [ ] 各セクションに「So What?」が 1〜3 行
- [ ] 事実主張ごとに出典 (URL / 発表元 / 年)
- [ ] テンション / 矛盾セクションがある
- [ ] 仮説の種が 3〜5 個 (根拠 / 想定機会 / 検証方法付き)
- [ ] 未解決の問いが明示されている
- [ ] HTML が単一ファイルで自己完結

## カスタマイズ

- **業界特有の観点を追加したい** → `templates/framework.md` に列を足す
- **レポートのデザインを変えたい** → `templates/report.html` の `<style>` を編集
- **リサーチプロセスを変えたい** → `.claude/skills/research-brief/SKILL.md` を編集
- **人格・基準を変えたい** → `CLAUDE.md` を編集

## ⚠️ 複数クライアントを扱う際の注意

このエージェントは複数案件で使われる前提で設計されています。
**クライアントの情報を混ぜないために以下を守ってください**:

1. **1 セッション = 1 クライアント = 1 案件**。
   新しい案件に取り掛かるときは、**必ず新しい Claude Code セッションを開く**。
   同じセッション内で別クライアントのブリーフを渡すのは厳禁。
2. **生成レポートは git にコミットされません** (`.gitignore` で `reports/*.html` を除外)。
   共有する場合は HTML ファイルを直接渡してください。
3. **過去の他案件レポートを Claude に読ませない**。Claude 側にも「`reports/` を Read しない」ルールを入れていますが、
   ユーザーから「過去の〇〇案件を参考に」と促すのも避けてください。
4. **NDA / 未発表情報** を含むブリーフは、Claude が Web 検索に投げる前にユーザー確認します。
   固有名詞を伏せた表現でリサーチが必要な場合はその旨を最初に伝えてください。
5. **ファイル命名**: `reports/<YYYYMMDD>-<クライアント>-<案件名>.html` でクライアント名を必ず含めます。

## 機微情報の扱い

NDA や未公開情報を含むブリーフは、外部 API (Web 検索など) に投げる前に
Claude がユーザーに確認します。判断に迷う情報があれば質問してください。
