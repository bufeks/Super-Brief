# Research Brief — クイック・キックオフ・ブリーフ整理フロー

Super-Brief の役割は **チーム即始動のためのクイック整理**。
**深い吟味は Agent-Teams 側で行われる前提**で、ここでは 30 分以内に
チームが議論を始められる情報整理を HTML レポートとして吐き出す。

## 起動条件

以下のいずれかで起動:

1. ユーザーが `/research-brief` を呼んだ
2. ブリーフ的ファイル (`.pdf`, `.pptx`, `.docx`, `.txt`, `.md`) を添付した
3. 「ブリーフを読んで」「リサーチして」「調べて」と頼まれた

## 🚨 失敗モードに注意

- ❌ ブリーフを要約だけチャットに返して終わる
- ❌ 「進めていいですか?」と途中で確認する (Step 0 以外は止まらない)
- ❌ サブエージェントを逐次起動 (必ず 1 メッセージで並列)
- ❌ 社会の空気 / 生活者 / カルチャーを **自分で深堀りする** (Agent-Teams 領域)
- ❌ HTML レポートを書かずにチャットだけで返す
- ❌ 下部 Sources セクションを作る (出典は引用箇所に `.src-inline` で添える)
- ❌ Creative Brief の Tagline / Key Visual / Catch Copy を埋める (別チーム領域)

## 全体フロー (7 ステップ)

```
[0] 分離チェック  クライアント混在の有無を確認 / 機微情報の有無 (1 ターンで完結)
[1] 抽出         scripts/extract.py でテキスト化
[2] デコード + 草案 要約 / 矛盾 / 不明点 / 質問 / Creative Brief 草案 (ブリーフ由来部分)
[3] 並列クイック   サブエージェント 2 個 (競合 + 市場) を 1 メッセージで並列起動
[4] 統合         Creative Brief 完成 + 可能性 + Agent-Teams 引継ぎリスト
[5] HTML 生成    reports/<日付>-<クライアント>-<案件>.html
[6] 自己レビュー  品質基準 → 不足補正 → SendUserFile 納品
```

---

## Step 0: 分離チェック

`CLAUDE.md` の「クライアント情報の分離」ルールに従い、リサーチに入る前に以下を **1 ターンで** 確認:

1. **このセッションで他クライアント案件を既に扱っていないか** 自分の文脈を確認
   - 既にあれば作業を中止し、「新しい Claude Code セッションで開き直してください」と返す
2. **`reports/` の中身は Read しない**
3. **クライアント名・案件名をユーザーに確認** (ブリーフから読み取れない場合)
4. **NDA / 機微情報の有無確認** — 含む場合は Web 検索クエリにそれを使わないことを宣言

このチェックが終わるまで Step 1 に進まない。

---

## Step 1: 抽出

```bash
python3 scripts/extract.py /absolute/path/to/brief.pdf
```

- 抽出失敗 → 別形式での提供を依頼
- 抽出結果が 200 文字未満 → OCR 必要の可能性を伝える
- テキストで貼られた場合はそのまま使う
- **抽出テキストをファイルに保存しない** (リポジトリ汚染防止)
- **ブリーフ入力をリポジトリ内にコピーしない** — 添付された一時パス (`/tmp/...`) をそのまま使う

---

## Step 2: デコード + 草案作成

ブリーフを読み、以下を素早く構造化する。HTML の **セクション 0〜4** に対応:

### Creative Brief 22 項目 (ブリーフから埋まる部分)

`templates/framework.md` の領域 0 を参照。**この段階では「ブリーフ記載」項目だけ充足**:
- Brand Name / Slogan / Promise / Business Goal / Ad Role / Problem / Competitor (記載分) / Target (記載分) / Tone (記載分) / Campaign Concept (記載分)
- それ以外は Step 3 (リサーチ) で埋めるか `[要確認]` のまま

### 要約 (HTML セクション 1)
- 30 秒で読める 3〜5 行
- 「何を売る / 誰に / いつどこで / なぜ今 / 難所」の構成

### 矛盾・課題 (HTML セクション 2)
- ブリーフ内の不整合 (2〜3 個)
- 実現性の疑問
- 優先順位の競合
- リスクの当たり (規制 / 炎上類例) ※深掘りは Agent-Teams へ

### 不明点 (HTML セクション 3)
- ブリーフに書かれておらず、リサーチでも確定できない情報
- 各不明点に「なぜ必要 / 誰に確認」を添える

### クライアントへの質問 (HTML セクション 4)
- **最大 7 個** に絞る (それ以上はクライアントの負担)
- 重要度の高い順
- 各質問に「なぜ聞きたい (プランへの効き方)」を添える

---

## Step 3: 並列クイック・リサーチ (★重要)

**サブエージェントは 2 個に絞る**。1 つのメッセージで複数の `Agent` ツール呼び出しを送る。

### サブエージェントへの指示で守ること (情報混在防止)
- 現案件のブリーフ抽出テキスト **だけ** を context として渡す
- 過去案件の固有名詞は絶対に書かない (検索クエリも)
- NDA / 未発表情報が含まれる場合、その固有名詞を Web に投げないよう明示

### 競合クイックエージェント
```
あなたは広告会社のクイック・リサーチャー。出典 URL 必須、2000 字以内、セクション分けて返す。
- 案件: <案件名・カテゴリー>
- 競合: <ブリーフ抽出から + 自分で発見>
- 各社の直近 12〜24 ヶ月の主要キャンペーン (キーメッセージ / 媒体)
- ポジショニング比較 (機能 / 情緒 / 価格 / 体験)
- 業界全体で使い古された訴求 (避けるべき表現)
深い競合分析や歴史は不要 — Agent-Teams 側で扱うため。
```

### 市場クイックエージェント
```
あなたは広告会社のクイック・リサーチャー。出典 URL 必須、2000 字以内、セクション分けて返す。
- カテゴリーの市場規模 (直近データ、発表元、年)
- 成長率・直近トレンド
- 構造変化 (チャネルシフト / 価格帯シフト)
- 関連規制 (景表法 / 薬機法 / 食品表示など該当のみ)
- 一次ソース優先 (政府統計 / 業界団体 / 上場企業 IR)
深い業界レポート読解は不要 — Agent-Teams 側で扱うため。
```

**並列実行のコツ**:
- 2 エージェント分の `Agent` ツール呼び出しを 1 メッセージにまとめる
- 各エージェントへ渡す context は最小限 (ブリーフ抽出 + 案件名)
- 各エージェントの返答は要点だけ保持し、生のログは破棄

---

## Step 4: 統合

サブエージェント結果を読み、以下を作る:

1. **Creative Brief 22 項目を完成** — リサーチ結果を該当セルに反映、不明は `[要確認]`
   - 18〜20 (Tagline / Key Visual / Catch Copy) は **完成形を書かない**、方向性のヒントのみ
2. **検討すべき可能性** (HTML セクション 7) — 3〜5 個
   - 各「可能性」に: 根拠 (ブリーフ) / 根拠 (外部) / 想定機会 / 検証方法
3. **Agent-Teams への引継ぎリスト** (HTML セクション 8)
   - 領域別 (社会の空気 / 生活者 / ブランド / カルチャー / 機会領域 / リスク詳細)
   - 各領域に「依頼内容 / 優先度」

---

## Step 5: HTML レポート生成

`templates/report.html` を参考に、**単一の HTML ファイル** を以下のパスに書き出す:

```
reports/<YYYYMMDD>-<クライアントスラッグ>-<案件名スラッグ>.html
```

例: `reports/20260522-acme-japan-natsu-no-yoru-craft-beer.html`

- **クライアント名を必ずファイル名に含める** (情報混在防止)
- スラッグは英小文字 / ハイフン / 数字のみ。日本語はローマ字化
- 既に同名ファイルがある場合は上書きしない — `-v2` 等を末尾に
- CSS は HTML 内 `<style>` に埋め込み、外部依存ゼロ
- セクション構成は `templates/report.html` の **8 セクション順** を守る:
  ```
  0. Creative Brief (22 項目)
  1. 要約 (30 秒で読む全体像)
  2. 矛盾・課題
  3. 不明点
  4. クライアントへの質問
  5. 競合情報 (クイック)
  6. 市場情報 (クイック)
  7. 検討すべき可能性
  8. Agent-Teams への引継ぎ
  ```
- **末尾 Sources セクションは作らない** — 出典は引用箇所に `.src-inline` で添える
- 文字コードは UTF-8、印刷でも崩れないよう `@media print` のスタイルを含める

### ブリーフ由来と外部由来の分離 (必須)

| 種類 | HTML 記法 |
|---|---|
| ブリーフ逐語引用 | `<blockquote class="brief-quote">...</blockquote>` |
| ブリーフ由来 (ブロック) | `<div class="from-brief">...</div>` |
| ブリーフ由来 (インライン) | `<span class="tag-brief">ブリーフ</span> 文章...` |
| 外部リサーチ由来 | 通常文 + `<span class="src-inline"><a href="URL">発表元</a> (年)</span>` を引用箇所のすぐ後ろに |
| リサーチャー解釈 | `<span class="tag-external">解釈</span>` タグ + So What / Tension カラウト |

- Creative Brief (Step 4) は混在。各セル内で出所をタグで区別
- 競合 / 市場 は基本的に外部リサーチ → 直後に `.src-inline`
- 可能性は「根拠 (ブリーフ)」「根拠 (外部)」の 2 行に分けて書く

ファイルを書き終わったら、絶対パスを SendUserFile でユーザーに送る。

---

## Step 6: 自己レビュー

レポートを書き終えた後、**自分で再度開いて以下をチェック**:

- [ ] CLAUDE.md の品質基準を満たすか
- [ ] **Creative Brief 22 項目** がレポート最上段、全項目記入 (不明は `[要確認]`)
- [ ] Creative Brief の 18〜20 (Tagline / Key Visual / Catch Copy) は方向性のみで完成形を書いていないか
- [ ] **8 セクションがこの順で揃っている** (Creative Brief → 要約 → 矛盾・課題 → 不明点 → 質問 → 競合 → 市場 → 可能性 → Agent-Teams 引継ぎ)
- [ ] **ブリーフ由来 / 外部由来 / 解釈** が視覚的に区別されているか
- [ ] 外部出典が引用箇所のすぐ後ろに小さく添えられているか
- [ ] 可能性が「根拠 (ブリーフ)」「根拠 (外部)」の 2 行に分かれているか
- [ ] 質問が **7 個以内** に絞られているか
- [ ] Agent-Teams 引継ぎセクションに領域別の依頼が書かれているか
- [ ] 「らしい」「と言われている」など曖昧表現が無いか
- [ ] HTML が外部 CSS/JS なしで自己完結しているか

不足があれば即座に修正して再書き出し。

## ユーザーへの最終納品メッセージ

レポート生成後、ユーザーへ以下を伝える:

1. レポートの HTML パス (SendUserFile で添付)
2. **矛盾・課題のうち最重要 3 個** (チーム議論の入口)
3. **クライアントへの質問 (聞きたい順 3 個)**
4. **Agent-Teams へ最優先で渡すべき領域 1〜2 個**

例:
```
✅ クイック・キックオフをまとめました → reports/20260522-xxx.html

最重要の議論論点:
1. ブリーフは「20代女性」と書きつつ実は M1F1 全体を狙っている可能性
2. 予算 1.5 億 vs. 媒体指定 (TV メイン) が両立しないのでは
3. ローンチ日 8/1 がカテゴリー繁忙期と被る

クライアントに聞きたい (順位順):
1. ターゲット「20代女性」のコア層は?
2. KPI は認知 / 売上 / ロイヤルティどれが最優先?
3. 競合 X 社の去年キャンペーンとの被りはどう考えていますか?

Agent-Teams へ最優先で:
- 生活者インサイト (N1 5〜10 件をターゲット拡張仮説と合わせて検証)
```
