# Research Brief — 広告ブリーフ・リサーチフロー

このスキルは Super-Brief リポジトリの中核フローです。
`CLAUDE.md` の人格定義と品質基準を前提として、以下のステップを実行します。

## 起動条件

以下のいずれかが満たされたら、このフローを実行します:

1. ユーザーが `/research-brief` を明示的に呼んだ
2. ユーザーがブリーフらしきファイル (`.pdf`, `.pptx`, `.docx`, `.txt`, `.md`) を添付した
3. ユーザーが「ブリーフを読んで」「リサーチして」のような依頼をした

## 🚨 失敗モードに注意

このフローでよくある失敗:

- **❌ ブリーフを要約して終わる** — それは Step 2 の一部に過ぎない。Step 7 まで通すのがゴール。
- **❌ 「進めてよいですか?」と途中で確認する** — Step 0 (機微情報・クライアント名) 以外は止まらない。
- **❌ サブエージェントを並列起動せず逐次で動かす** — 効率と独立性のため必ず 1 メッセージで並走。
- **❌ HTML レポートを書かずにチャットだけで返す** — 納品物は `reports/` 配下の HTML ファイル。
- **❌ Step 6 のあと自己レビュー (Step 7) を省略する** — 品質基準 8 項目を必ずチェック。

## 全体フロー (8 ステップ)

```
[0] 分離チェック  クライアント混在の有無を確認 → クリーンスタート
[1] 抽出      ブリーフ → テキスト
[2] デコード  表層課題 / 真因仮説 / 違和感 / 残存する問い
[3] 計画      10 領域からこの案件で重点的に見るべき領域を選定
[4] 並列リサーチ  Sub-agent を使って 3〜5 領域を並列実行
[5] 統合      横断テンション・矛盾の発見、So What の抽出
[6] レポート生成   templates/report.html の構造で HTML を生成
[7] 自己レビュー  品質基準チェック → 不足を補完 → 納品
```

---

## Step 0: 分離チェック (必ず最初に実行)

`CLAUDE.md` の「クライアント情報の分離」ルールに従い、**リサーチに入る前** に以下を必ず確認する。

1. **このセッションで既に別クライアントの案件を扱っていないか** 自分の文脈を確認する。
   - 既に別案件があれば、作業を中止して以下のメッセージを返す:
   ```
   ⚠️ このセッションで既に別案件 (XXX) のリサーチを行っています。
   情報混在を防ぐため、新しい Claude Code セッションで開き直してください。
   ```
2. **`reports/` ディレクトリの中身は Read しない**。
   テンプレート参照は `templates/report.html` だけから行う。
3. **クライアント名・案件名をユーザーに確認** (ブリーフから読み取れない場合)。
   ファイル命名と Step 6 のスラッグ化で使う。
4. **NDA / 未発表情報の確認**: ブリーフが機微情報を含むか聞く。
   含む場合は、Web 検索クエリにそれを使わないことを宣言する。

このチェックが終わるまで Step 1 に進まない。

## Step 1: 抽出

入力ファイルが添付されている場合:

```bash
python3 scripts/extract.py /absolute/path/to/brief.pdf
```

- 抽出に失敗したら、ユーザーに別形式での提供を依頼
- 抽出結果が極端に短い (200 文字未満) 場合、OCR が必要な可能性を伝える
- ブリーフがテキストで貼られた場合はそのまま使う

抽出したテキストはチャットには長文を貼らず、内部メモとして扱う。
**抽出テキストをファイルに保存しない** (リポジトリ汚染防止)。
ブリーフ入力ファイル自体もリポジトリの中にコピーしない — 添付された一時パス (`/tmp/...` 等) をそのまま使う。

---

## Step 2: ブリーフ・デコード

抽出テキストを読み、以下を構造化する。**この段階で 1 度ユーザーに確認しない。先に進む。**
ただし最終レポートに残す。

```yaml
案件名: 
クライアント: 
カテゴリー: 
表層課題:        # ブリーフに書いてある課題そのまま
推測される真因:  # なぜその課題が生じているかの仮説 2〜3 個
ターゲット:
  ブリーフ記載: 
  追加で見るべき層: 
KPI/成功指標:
予算/期間/媒体制約:
過去施策の言及:
違和感・疑問:    # 必ず 3 個以上挙げる
追加で欲しい情報: # ユーザーに後で聞くと良いこと
```

---

## Step 3: リサーチ計画

10 領域から、この案件で **最低 6 領域** を選び、各領域で何を調べるか宣言する。
ユーザー向けにも 5〜10 行でリサーチ計画を投稿する。

```
リサーチ計画:
- 競合状況: 〇〇社、〇〇社のキャンペーン3年分、新規参入の有無
- 市場・カテゴリー: 市場規模、成長率、〇〇規制の動向
- 社会の空気: ターゲット世代の価値観調査、関連バズ
- 生活者インサイト: SNS でのカテゴリー言及、N1 引用 10件
- カルチャー文脈: 関連サブカルチャー、ミーム
- リスク・地雷: 過去の炎上事例、規制
```

---

## Step 4: 並列リサーチ (重要)

**Sub-agent を並列に起動して context を節約する**。
1 つのメッセージで複数の `Agent` ツール呼び出しを送る。

**サブエージェントへの指示で守ること (情報混在防止)**:
- 現案件のブリーフ抽出テキスト **だけ** を context として渡す
- 過去の案件名、他クライアント名を絶対に書かない (検索クエリにも入れない)
- 「前回の〇〇案件では」のような参照は禁止
- NDA / 未発表情報が含まれる場合、その固有名詞を Web 検索に投げないよう明示
- 各サブエージェントはこのセッションの context を共有しないので、必要な情報は毎回最小限を再記述する

各 Sub-agent への指示テンプレ:

### 競合リサーチエージェント
```
あなたは広告会社のリサーチャーです。以下を調べてレポート:
- 案件: <案件名・カテゴリー>
- 競合: <ブリーフから抽出 + 自分で発見した競合>
- 各社の直近 2 年の主要キャンペーン (キーメッセージ / クリエイティブ方向 / 媒体)
- ポジショニング比較 (機能訴求 / 情緒訴求 / 価格 / 体験)
- 新規参入や脅威となる隣接カテゴリー
出典 URL を必ず付ける。3000 字以内でセクション分けして返す。
```

### 市場・カテゴリーエージェント
```
- カテゴリーの市場規模 (直近データ、発表元、年)
- 成長率と直近のトレンド
- 構造変化 (チャネルシフト、価格帯シフト、ユーザー層の変化)
- 関連する規制・自主規制
- 一次ソース (政府統計、業界団体、上場企業 IR) を優先
3000 字以内、出典必須。
```

### 社会の空気 / 生活者エージェント
```
- ターゲット層の最新の価値観調査 (博報堂生活総研、電通、リクルート等)
- カテゴリーに関する SNS (X, Instagram, TikTok, Threads) での会話傾向
- N1 の声を 5〜10 件、要旨で引用 (個人特定可能情報は除く)
- 関連するバズ / 流行語 / ミーム
- 生活者の Tension (期待と現実のズレ)
出典 URL 必須、3000 字以内。
```

### カルチャー / リスクエージェント
```
- 案件カテゴリーに関連するサブカルチャー、コミュニティ
- 影響力のある人物 (タレント、クリエイター、KOL)
- 過去 3 年の炎上事例 (案件カテゴリー / 競合 / 類似訴求)
- 規制チェック (景表法 / 薬機法 / 食品表示など該当する範囲)
- ブランドセーフティ上の地雷
3000 字以内、出典必須。
```

### 過去施策・ブランド現在地エージェント (該当する場合)
```
- ブランド <X> の過去 3〜5 年の主要キャンペーン
- 想起されている価値、トーン
- 直近の評判 (好意 / 否定)
- 認知・好意度の公開データがあれば
出典必須、2000 字以内。
```

並列実行のコツ:
- 各エージェントには **Explore** または **general-purpose** を使い分ける。
- 同じ Web を重複検索しないよう、互いに切り分けた領域を渡す。
- 各エージェントの返答は要約を保持し、生のログは破棄。

---

## Step 5: 統合

サブエージェントの結果を読み、以下を作る:

1. **領域横断テンション** — 「市場は伸びているが SNS は冷めている」のような複数領域にまたがる発見
2. **矛盾の整理** — ブリーフの想定と現実のギャップ
3. **So What 抽出** — 各セクションに 1〜3 行の含意
4. **仮説の種** — 3〜5 個。各仮説に「根拠 / 想定機会 / 検証方法」を添える
5. **未解決の問い** — 追加調査やクライアントへの確認が必要な論点

## Step 5.5: Creative Brief 確定 (★必須)

`templates/framework.md` の「領域 0」に定義された **22 項目** を全部埋める。
これが HTML レポート最上段の「セクション 0」として配置される、**プランナーが最初に見るもの**。

### 必須 22 項目 (左カラム: ブランド軸 10 + 右カラム: ターゲット・実装 12)

```
[ブランド軸]
1.  Brand Name
2.  Brand Philosophy / Purpose
3.  Brand Slogan
4.  Brand Promise
5.  Business Goal
6.  Ad Role
7.  Problem
8.  Competitor
9.  Unique Selling Proposition
10. Fact

[ターゲット・実装]
11. Target
12. Target Insight
13. Social Insight
14. Before Perception
15. After Perception
16. Tone & Manner
17. Campaign Concept
18. Campaign Tagline       ※ プランナー領域 → 方向性のみ
19. Key Visual             ※ クリエイティブ領域 → 方向性のみ
20. Catch Copy             ※ コピーライター領域 → 方向性のみ
```

### 充足ルール

- 各セルは 1〜3 行で簡潔に記入
- 各セル内で出所を区別:
  - ブリーフ記載 → `.from-brief` 黄色枠 or `<span class="tag-brief">ブリーフ</span>` インライン
  - 外部リサーチ → 通常文 + `<span class="src-inline">` で発表元・年・URL を引用直後に添える
  - 解釈 → `<span class="tag-external">解釈</span>` タグ付き
- ブリーフからも外部からも確定できない場合は `[要確認]` と書き、Step 7 の「未解決の問い」に追加
- 18〜20 (Tagline / Key Visual / Catch Copy) は別チームの仕事なので、**完成形を絶対に書かない**。方向性のヒントまで。

## Step 6: HTML レポート生成

`templates/report.html` を参考に、**単一の HTML ファイル** を以下のパスに書き出す:

```
reports/<YYYYMMDD>-<クライアントスラッグ>-<案件名スラッグ>.html
```

例: `reports/20260521-acme-japan-natsu-no-yoru-craft-beer.html`

- **クライアント名を必ずファイル名に含める** (情報混在防止のため省略不可)
- スラッグは英小文字 / ハイフン / 数字のみ。日本語はローマ字化。
- 既に同名ファイルがある場合は **上書きしない** — `-v2` 等を末尾に付ける

- CSS は HTML 内 `<style>` に埋め込み、外部依存ゼロ。
- セクション構成は `templates/report.html` の通り (Overview / Decode / 10領域 / Tension / Hypothesis / Open Questions)。**末尾 Sources セクションは作らない** — 出典は引用箇所に小さく添える方式に統一
- 文字コードは UTF-8、印刷でも崩れないよう `@media print` のスタイルを含める

### ブリーフ由来と外部由来の分離 (必須)

`templates/report.html` の CSS を使って、すべての事実主張を以下のいずれかで明示する:

| 種類 | HTML 記法 |
|---|---|
| ブリーフ逐語引用 | `<blockquote class="brief-quote">...</blockquote>` |
| ブリーフ由来の主張 (ブロック) | `<div class="from-brief">...</div>` |
| ブリーフ由来 (インライン) | `<span class="tag-brief">ブリーフ</span> 文章...` |
| 外部リサーチ由来 | 通常文 + `<span class="src-inline"><a href="URL">発表元</a> (年)</span>` を引用箇所のすぐ後ろに |
| リサーチャー解釈 | `<span class="tag-external">解釈</span>` タグ + So What / Tension カラウト |

- Decode セクション (Step 2) はほぼ全てブリーフ由来 → 黄色枠で。違和感・真因仮説は「解釈」タグ付き。
- 競合 / 市場 / 社会 / 生活者 / カルチャー / リスク は基本的に外部リサーチ → 番号付き出典。
- 仮説の種は「根拠 (ブリーフ)」と「根拠 (外部)」の **2 行** に分けて書く。混ぜない。
- **下部 Sources セクションは作らない**。外部出典は引用したその場で `.src-inline` (短い注記) または `.src-line` (ブロックの下に新行) を使って小さく添える。読み手が「どの主張がどのソースか」を視線移動なしで確認できることを優先。

ファイルを書き終わったら、絶対パスを SendUserFile でユーザーに送る。

## Step 7: 自己レビュー

レポートを書き終えた後、**自分で再度開いて以下をチェック**:

- [ ] CLAUDE.md の品質基準を満たすか
- [ ] **Creative Brief 22 項目** がレポート最上段に配置され、全項目記入されているか (不明は `[要確認]`)
- [ ] Creative Brief の 18〜20 (Tagline / Key Visual / Catch Copy) は方向性のみで完成形を書いていないか
- [ ] **ブリーフ由来 / 外部由来 / 解釈 の 3 区分が視覚的に区別** されているか (黄色枠 / 通常 + 番号引用 / 解釈タグ)
- [ ] ブリーフ由来と外部由来が **1 文の中で混ざっていない** か
- [ ] 外部出典が **引用箇所のすぐ後ろ** に小さく添えられているか (下部 Sources セクションは作っていないか)
- [ ] 仮説の種が「根拠 (ブリーフ)」「根拠 (外部)」の 2 行に分かれているか
- [ ] 外部出典 URL が事実主張ごとに付いているか
- [ ] 「らしい」「と言われている」など出典なしの曖昧表現が無いか
- [ ] 仮説の種が広告プランの提示になっていないか (プランは別チームの仕事)
- [ ] HTML が外部 CSS/JS なしで自己完結しているか
- [ ] 古い統計 (5 年以上前) を最新扱いしていないか

不足があれば即座に修正して再書き出し。

## ユーザーへの最終納品メッセージ

レポート生成後、ユーザーへ以下を伝える:

1. レポートの HTML パス (SendUserFile で添付)
2. 10 領域のうち何をカバーしたか (1 行サマリ)
3. **特に重要な発見 3 個** を短く
4. **未解決の問い** (ユーザーに確認したい論点) を箇条書きで

例:
```
✅ レポートを reports/20260521-xxx.html に出力しました。

カバー領域: 競合 / 市場 / 社会の空気 / 生活者 / カルチャー / リスク (6 領域)

特に重要な発見:
1. 市場は CAGR +12% で成長中だが、SNS では「またこのカテゴリーか」という飽き
2. 競合 A 社が 2025 年から訴求を機能 → 情緒にシフト、3 社が追随
3. 過去 12 ヶ月で類似訴求の炎上 2 件 (添加物 / 環境負荷)

ご確認いただきたい論点:
- ターゲットの「20代女性」は M1F1 全体ですか、それともコア (24-27) ですか?
- 過去施策のうち最も成果が出たものを 1 つ教えていただけますか?
```
